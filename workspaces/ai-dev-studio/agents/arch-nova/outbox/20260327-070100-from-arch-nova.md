# Design: Per-Agent Executor Selection

## Summary

Allow each agent to declare which execution backend it uses via an `executor` field in `agent.json`. `AgentRunnerService` resolves the correct `IAgentExecutor` at launch time.

## 1. DI Registration Changes in `Program.cs`

Replace the single `IAgentExecutor` registration with keyed registrations for each executor. Use `IEnumerable<IAgentExecutor>` so the runner can resolve all of them at startup.

```csharp
// REMOVE this line:
// builder.Services.AddSingleton<IAgentExecutor, ClaudeAgentExecutor>();

// ADD:
builder.Services.AddSingleton<IAgentExecutor, ClaudeAgentExecutor>();
// Future executors register the same way:
// builder.Services.AddSingleton<IAgentExecutor, OllamaAgentExecutor>();
```

The key change is **not** in `Program.cs` registration (the above already works for `IEnumerable<>` injection) — it's in how `AgentRunnerService` consumes the executors.

Alternatively, if you want explicit keyed services (.NET 8+):

```csharp
builder.Services.AddKeyedSingleton<IAgentExecutor, ClaudeAgentExecutor>("claude");
// builder.Services.AddKeyedSingleton<IAgentExecutor, OllamaAgentExecutor>("ollama");
```

**Recommendation**: Use the simpler `IEnumerable<IAgentExecutor>` approach. It requires no attribute annotations and adding a new executor is just one line. Keyed services add complexity without clear benefit here since we already have `Name` on the interface.

## 2. `AgentRunnerService` Constructor Change

Change the constructor to accept `IEnumerable<IAgentExecutor>` and build an internal lookup dictionary:

```csharp
public class AgentRunnerService(
    WorkspaceService workspace,
    StudioSettingsService settings,
    IEnumerable<IAgentExecutor> executors,
    ILogger<AgentRunnerService> logger)
{
    private readonly Dictionary<string, IAgentExecutor> _executors =
        executors.ToDictionary(e => e.Name, StringComparer.OrdinalIgnoreCase);
```

## 3. Fail-Fast vs. Fallback

**Recommendation: Fail fast.** Throw `InvalidOperationException` if no matching executor is found.

Rationale:
- A silent fallback to `"claude"` masks configuration errors. If someone sets `"executor": "ollama"` but the `OllamaAgentExecutor` isn't registered, they'd get a Claude session with no indication anything is wrong.
- An agent session costs real time and money. Running the wrong backend wastes both.
- The error surfaces at launch time (not deep in a session), so it's easy to diagnose.

In `RunSessionAsync`, after reading `agent.json`:

```csharp
var executorName = agentInfo?.GetValueOrDefault("executor")?.ToString() ?? "claude";
if (!_executors.TryGetValue(executorName, out var selectedExecutor))
{
    throw new InvalidOperationException(
        $"No executor registered with name '{executorName}'. " +
        $"Available: [{string.Join(", ", _executors.Keys)}]");
}

var psi = selectedExecutor.BuildProcessStartInfo(agentDir, modelId, AgentPrompt);
```

The default value `"claude"` preserves backward compatibility for agents that don't specify an executor.

## 4. `AgentInfo` Changes

Add an `Executor` property to `AgentInfo`:

```csharp
public class AgentInfo
{
    public string Slug { get; set; } = string.Empty;
    public string Name { get; set; } = string.Empty;
    public string Role { get; set; } = string.Empty;
    public string Model { get; set; } = "sonnet";
    public string Executor { get; set; } = "claude";
    public string Status { get; set; } = "idle";
    public string Description { get; set; } = string.Empty;
    public string? LastRunAt { get; set; }
    public int InboxCount { get; set; }
}
```

This is populated by the existing JSON deserialization of `agent.json` — no special reader needed since `System.Text.Json` will bind the `executor` field automatically (camelCase policy is already in use).

## 5. `agent.json` Schema Conventions

- **Field name**: `executor` (camelCase, consistent with `model`, `status`, etc.)
- **Type**: `string`
- **Default**: `"claude"` — the field is optional. Omitting it means Claude.
- **Valid values**: Must match an `IAgentExecutor.Name` exactly (case-insensitive matching recommended).
- **Examples**:

```json
{ "executor": "claude" }
{ "executor": "ollama" }
```

The field sits at the top level of `agent.json`, alongside `slug`, `model`, `role`, etc.

## Trade-offs

| Approach | Pros | Cons |
|---|---|---|
| `IEnumerable<IAgentExecutor>` (recommended) | Simple, no new DI concepts, one line to add executor | Dictionary built at first use |
| Keyed services | Framework-native lookup | Requires `[FromKeyedServices]` attributes, .NET 8+ only |
| Factory pattern | Maximum flexibility | Over-engineered for string-keyed lookup |

## Implementation Checklist

1. Add `Executor` property to `AgentInfo` (default `"claude"`)
2. Change `AgentRunnerService` constructor: `IAgentExecutor` → `IEnumerable<IAgentExecutor>`
3. Build `_executors` dictionary in constructor
4. In `RunSessionAsync`: read `executor` from `agentInfo`, resolve from dictionary, fail fast if missing
5. Remove dead `BoardService` registration on line 15 of `Program.cs`
6. No changes to `Program.cs` DI registrations needed (single `AddSingleton<IAgentExecutor, T>` already works with `IEnumerable<>`)
